### 얼굴 매칭 → DB → WebSocket → NotificationBell 끝까지 연결

핵심 가치 흐름이 처음으로 한 번에 동작하도록 다섯 가지 결함을 잡았다.

- **OpenCV RTSP open 한계 우회**: `opencv-python-headless 4.11`이 일부 카메라(Tapo C500)의 RTSP를 안정적으로 못 여는 문제 발견. `frame_capture`를 `ffmpeg subprocess + image2pipe(mjpeg) + -stimeout 8000000`으로 교체. backend 컨테이너에 `ffmpeg` 영구 추가(`Dockerfile.gpu`).
- **frame_capture가 카메라 직접 RTSP 대신 go2rtc 릴레이(`rtsp://localhost:8554/{camera_id}`)를** 기본 사용하도록 토글(`frame_capture_use_relay`) 도입. 라이브뷰가 동작하는 카메라는 frame_capture도 무조건 동작.
- **DB `cameras.status`(항상 OFFLINE)에 의존하던** worker 필터를 제거 → `camera_status_resolver`로 RTSP 도달성 + go2rtc 등록 여부 기반 ONLINE 판정.
- **recognition_log_usecase가 임계치 미만이어도** 항상 `top_score`를 confidence로 기록하고 INFO 로그(`Face search: top_score=...`)를 남겨 진단을 쉽게 했다. `.env`에서 `RECOGNITION_THRESHOLD=0.45`, `MOTION_GATE_ENABLED=false`로 검증 운영값 조정.
- **트랜잭션 롤백 결함 발견**: face_recognition_bootstrap의 별도 session 컨텍스트에 `UserORM`이 import 안 되어 SQLAlchemy 메타데이터에서 `danger_events.resolved_by → users` FK 매핑 실패 → INSERT 거부 → 같은 트랜잭션의 recognition_log도 함께 롤백되던 문제. `infrastructure/database/orm_models.py`에 모든 ORM을 일괄 import하는 registry 추가. 다른 도메인 ORM 누락 함정도 함께 차단.
- **frontend NotificationBell이 빈 상태로 남던 결함 진단**: `JotaiProvider`에 명시적 store 인스턴스(`infrastructure/jotai/store.ts`)를 부여해 default store 의존을 제거했고, 그래도 안 떠 직접 ws 메시지 도달은 `wsClient` 직접 진단으로 확인 → 진짜 원인은 **`useNotification`이 atom을 상대경로(`../atoms/notificationStore`)로 import하고 다른 hook은 alias(`@/features/...`)로 import하던 path mismatch**였다. Turbopack이 두 경로를 다른 모듈 인스턴스로 처리해 atom이 두 개로 만들어졌고, ws 측 set과 hook 측 get이 서로 다른 atom을 보고 있었음. 모든 import를 alias로 통일하자 즉시 정상 표시.
- 검증: C500 앞 5초 → `recognition_logs`에 이승욱(confidence 0.738) 신규 + `danger_events` 신규 + ws broadcast `type=DANGER_EVENT sent=1/1` + NotificationBell에 `[MEDIUM] 얼굴 인식 — [얼굴 인식 알림] 얼굴 인식: 이승욱 (임직원), 신뢰도: 70.1%` 카드 노출까지 한 번에 통과.
